# Kaiten Landing Factory Skill

## 1. Назначение скилла

Используй этот скилл, когда нужно создать новый лендинг Kaiten для конкретной сферы, команды, отдела, бизнес-процесса или аудитории.

Примеры задач:

- «Сделай лендинг Kaiten для юридических отделов».
- «Собери лендинг Kaiten для e-commerce-команд».
- «Нужен лендинг для проектных офисов».
- «Сделай страницу Kaiten для строительного контроля».
- «Собери лендинг-решение для команд продаж».
- «Сделай лендинг для внутренней техподдержки».
- «Создай лендинг для финансового отдела».

Цель скилла — не подготовить рекомендации, не написать отдельное ТЗ и не описать будущую страницу. Цель — создать готовый лендинг, который можно передать разработчику для проверки и публикации.

Агент должен работать как полноценный контент-завод:

- исследователь;
- продуктовый маркетолог;
- редактор Kaiten;
- UX-проектировщик;
- дизайнер в рамках дизайн-системы Kaiten;
- SEO-специалист;
- фронтенд-сборщик;
- QA-редактор перед публикацией.

---

## 2. Что агент получает на входе

Минимальный вход:

- тема лендинга;
- аудитория или сфера, если она известна.

Пример:

«Kaiten для юридических отделов».

Дополнительные данные, если пользователь дал их:

- приоритетные фичи;
- ссылки на конкурентов;
- нужные кейсы;
- SEO-ключи;
- ограничения по структуре;
- требования к CTA;
- требования к дизайну;
- стек проекта;
- формат верстки;
- требования к аналитике;
- готовые шаблоны Kaiten;
- доступ к рабочему пространству Kaiten для создания шаблонов.

Если критически важных данных нет, задай короткие уточняющие вопросы. Если данных достаточно для разумного старта, не тормози задачу и делай аккуратные предположения. Все предположения вынеси в блок `needs_confirmation`.

---

## 3. Что агент должен отдать на выходе

Финальный результат — готовый лендинг под публикацию.

В результате должны быть:

- исследование аудитории;
- анализ конкурентов и лучших практик;
- список использованных источников;
- подбор релевантных кейсов Kaiten;
- SEO-ядро;
- SEO title;
- SEO description;
- человекопонятный URL;
- структура страницы;
- готовые тексты всех блоков;
- готовые визуалы или промпты/описания для генерации визуалов, если прямой генерации нет;
- готовая адаптивная верстка;
- CTA-кнопки;
- ссылки на кейсы;
- ссылки на шаблоны;
- ссылки на продуктовые страницы и базу знаний, если они нужны;
- alt-тексты для изображений;
- места под цели Яндекс Метрики;
- финальный чек-лист самопроверки;
- список вопросов, которые нужно подтвердить у человека, если такие есть.

Разработчик не должен собирать лендинг с нуля. Он получает готовую страницу и проверяет техническую совместимость с проектом.

## 4. Главные источники

Перед созданием лендинга агент обязан опираться на источники.

### 4.1. Редполитика Kaiten

Основной файл:

`Редполитика+Kaiten.md`

Используй его как главный источник по:

- tone of voice;
- принципам текста;
- названию Kaiten / Кайтен;
- описанию продукта;
- допустимым и недопустимым формулировкам;
- требованиям к тексту;
- оформлению;
- иллюстрациям;
- публикации.

Ключевые правила из редполитики:

- Kaiten говорит просто, полезно, кратко и уважительно.
- Kaiten — умный, но не занудный; уверенный, но не дерзкий; понятный, но экспертный; дружелюбный, но знает границы.
- Текст должен давать пользу, а не существовать ради текста.
- Не писать воду.
- Не использовать сложные конструкции без необходимости.
- Не ставить читателя в позицию дурака.
- Не быть снисходительным.
- Не давать ложных обещаний.
- Не писать лозунгами.
- Опирайся на факты, сценарии и реальные возможности продукта.

### 4.2. Основные публичные источники Kaiten

Используй эти источники для проверки продукта, структуры и референсов:

- Главная Kaiten: https://kaiten.ru/
- База знаний Kaiten: https://faq-ru.kaiten.site/
- Кейсы Kaiten: https://kaiten.ru/blog/tag/case/
- IT-команды: https://kaiten.ru/teams/kaiten-for-it
- Маркетинг: https://kaiten.ru/teams/kaiten-for-marketing
- HR: https://kaiten.ru/teams/kaiten-for-hr/
- Редакции: https://kaiten.ru/teams/kaiten-for-edit
- Инженеры и строители: https://kaiten.ru/teams/kaiten-for-engineer-and-constructing
- Бухгалтерия: https://kaiten.ru/teams/kaiten-for-accounting
- Веб-агентства: https://kaiten.ru/teams/kaiten-for-web-agency

### 4.3. Референсные ТЗ и лендинги

Используй приложенные ТЗ как референсы структуры, тона и логики блоков:

- Кайтен для саппорта.
- Кайтен для редакций.
- Кайтен для маркетинга.
- Кайтен для производств.
- Кайтен для канбан-команд.
- Кайтен для веб-агентств.
- Кайтен для бухгалтерии.
- Госкомпании и учреждения.

Не копируй текст дословно, если делаешь новый лендинг. Используй референсы как образец логики:

- hero показывает конкретный процесс;
- после hero есть короткие преимущества;
- кейсы подбираются по сфере или похожей задаче;
- фичи раскрываются через сценарии пользователя;
- документы и шаблоны — разные блоки;
- ближе к концу есть запуск за 30 минут;
- финальный блок закрывает страницу и ведет к CTA.

### 4.4. Конкуренты

Конкурентов использовать только для анализа:

- какие боли они подсвечивают;
- какие блоки используют;
- какие CTA ставят;
- какие возражения закрывают;
- как объясняют решение;
- какие best practices можно учесть.

Нельзя копировать тексты, структуру, визуальные решения, офферы и обещания конкурентов.

## 5. Как ссылаться на источники

Агент должен сохранять связь между решениями на странице и источниками.

В финальной выдаче добавь технический блок `source_map`, который не публикуется на лендинге, но остается для команды.

В `source_map` укажи:

- какие лендинги использованы как структурные референсы;
- какие страницы базы знаний использованы для проверки фичей;
- какие кейсы использованы;
- какие продуктовые страницы использованы;
- какие конкуренты изучены;
- какие утверждения требуют подтверждения.

На опубликованной странице не нужно ставить академические сноски. Но все продуктовые факты должны быть проверяемыми.

Если фича не подтверждена источником, не используй ее как факт. Вынеси в `needs_confirmation`.

## 6. Роль агента

При создании лендинга агент должен последовательно мыслить в нескольких ролях.

### 6.1. Руководитель или тимлид целевой команды

Думает о:

- прозрачности процессов;
- сроках;
- загрузке сотрудников;
- статусах;
- рисках;
- отчетности;
- ответственности;
- контроле без микроменеджмента;
- предсказуемости работы.

### 6.2. Сотрудник команды

Думает о:

- понятных задачах;
- приоритетах;
- сроках;
- документах;
- комментариях;
- отсутствии лишних переключений;
- понятных правилах процесса.

### 6.3. Смежная команда или внутренний заказчик

Думает о:

- передаче заявок;
- статусе запроса;
- понятном канале коммуникации;
- сохранении контекста;
- отсутствии потерянных договоренностей.

### 6.4. Руководитель бизнеса

Думает о:

- управляемости;
- скорости процессов;
- экономии времени на ручном контроле;
- отчетности;
- внедрении;
- безопасности;
- масштабировании;
- переносе данных из старой системы.

### 6.5. Редактор Kaiten

Проверяет:

- пользу;
- точность;
- простоту;
- уважительный тон;
- отсутствие лозунгов;
- отсутствие ложных обещаний;
- соответствие редполитике.

### 6.6. Дизайнер и фронтенд-сборщик

Делает страницу:

- визуально похожей на Kaiten;
- адаптивной;
- понятной;
- готовой к публикации;
- с интерфейсными визуалами под сферу.

## 7. Исследование аудитории

Перед созданием страницы проведи исследование.

Определи:

- кто основная аудитория;
- кто принимает решение;
- кто влияет на решение;
- какие процессы есть у команды;
- какие задачи повторяются каждый день, неделю, месяц;
- какие статусы важно видеть;
- где сейчас теряются задачи, документы, сроки и договоренности;
- какие инструменты команда может использовать сейчас;
- что раздражает руководителя;
- что раздражает сотрудников;
- какие риски возникают из-за непрозрачности;
- какие критерии выбора системы важны;
- какие возражения могут быть;
- какие фичи Kaiten закрывают эти задачи;
- какие кейсы могут подтвердить пользу.

Главный фокус: обычно решение принимает тимлид, руководитель отдела, руководитель направления или человек, которому нужна прозрачность процессов.

---

## 8. Как формулировать боли

Не иди через жесткий негатив. Не обвиняй команду и руководителя.

Формулируй через управляемое улучшение.

Плохо:

«Тимлид не видит, что делает команда».

Хорошо:

«Тимлид видит загрузку команды и статус задач без лишних созвонов».

Плохо:

«Задачи теряются в хаосе».

Хорошо:

«Задачи из чатов, почты и форм собираются на одной доске».

Плохо:

«Руководитель вынужден всех контролировать».

Хорошо:

«Руководитель видит процесс без микроменеджмента».

Плохо:

«Сотрудники забывают дедлайны».

Хорошо:

«Сроки подсвечиваются на доске, а команда видит приоритеты заранее».

---

## 9. Как писать

### 9.1. Не писать маркетинговые лозунги

Запрещены формулировки:

- «Выведите бизнес на новый уровень».
- «Революционное решение».
- «Лучший инструмент для всех задач».
- «Идеальная система».
- «Забудьте о хаосе навсегда».
- «Увеличьте эффективность в 10 раз».
- «Полный контроль над всем бизнесом».

Пиши конкретно:

- «Задачи, сроки и документы собраны в одном пространстве».
- «Руководитель видит статус каждого этапа без ручных отчетов».
- «Команда понимает, что в работе, что ждет согласования и где есть риск задержки».

### 9.2. Не давать ложных обещаний

Не обещай:

- гарантированный рост эффективности;
- мгновенное внедрение без участия команды;
- полное избавление от ошибок;
- автоматическое решение всех проблем управления;
- конкретные цифры без источника;
- импорт из конкретной системы, если он не подтвержден;
- функции, которые не проверены по базе знаний.

Пиши аккуратно:

- «Kaiten помогает сделать процесс прозрачнее».
- «Команда может настроить доски под свои этапы».
- «Руководитель получает больше данных для управления сроками и загрузкой».

### 9.3. Не писать языком фичей

Плохо:

«В Kaiten есть WIP-лимиты, swimlanes, блокировки, кастомные поля и диаграмма Ганта».

Хорошо:

«WIP-лимиты помогают не перегружать команду задачами. Дорожки показывают загрузку сотрудников или отделов. Блокировки подсвечивают задачи, которые остановились и мешают потоку».

Формула:

`фича → рабочий сценарий → польза для роли`

Пример:

Фича: карточка задачи.

Плохо:

«В Kaiten есть карточки задач».

Хорошо:

«Открыли карточку — видите задачу, ответственного, срок, документы, чек-лист и всю историю обсуждений. Не нужно искать контекст в почте, чатах и на диске».

## 10. Работа с англицизмами и иностранными словами

На опубликованных страницах Kaiten нужно избегать англицизмов и иностранных слов, если у них есть понятный русский аналог. Это правило относится к заголовкам, подзаголовкам, карточкам, кнопкам, подписям, alt-текстам, вопросам и ответам, SEO title и description.

Исключения:

- названия сервисов, продуктов, компаний и зарегистрированных товарных знаков: Kaiten, Jira, Trello, Notion, YouTrack, Asana, ClickUp;
- термины, у которых нет точного общеупотребительного русского аналога или которые закреплены в профессиональной среде: канбан, скрам;
- технические атрибуты в коде, служебные поля, имена классов и внутренние комментарии для разработчика;
- аббревиатуры, которые нужны для точности, если рядом есть русская расшифровка.

Если иностранное слово необходимо, при первом употреблении дай русский вариант и расшифровку в скобках. Дальше используй русский вариант.

Пример:

- отдел персонала (HR), дальше — отдел персонала;
- соглашение об уровне обслуживания (SLA), дальше — уровень обслуживания или SLA, если это устойчивый термин для аудитории;
- ключевые показатели эффективности (KPI), дальше — показатели;
- бизнес для бизнеса (B2B), дальше — B2B, если без аббревиатуры текст становится тяжелым;
- гибкие подходы (Agile), дальше — гибкие подходы;
- электронная коммерция (e-commerce), дальше — интернет-торговля или электронная коммерция.

Канбан и скрам писать по-русски. Не писать Kanban и Scrum в опубликованном тексте, если это не часть названия интеграции, документации или цитаты.

### 10.1. Словарь замен для публичных страниц

Перед публикацией проверь текст и замени англицизмы.

Используй такие варианты:

- landing page / landing / лендинг → страница, промостраница, страница решения;
- hero / hero-блок → первый экран;
- CTA → кнопка действия, кнопка, призыв к действию;
- FAQ → вопросы и ответы;
- feature / фича → возможность, функция, инструмент;
- use case → сценарий использования, рабочий сценарий;
- case / кейс → история клиента, пример внедрения, история использования;
- workflow → рабочий процесс;
- dashboard → панель отчетов, сводка, дашборд допустим только если аудитория точно так говорит;
- roadmap → дорожная карта;
- backlog → очередь задач, список задач, бэклог допустим для ИТ-аудитории;
- sprint → спринт допустим для ИТ и скрам-команд, иначе — рабочий цикл;
- deadline → срок;
- team lead / тимлид → руководитель команды, тимлид допустим для ИТ-аудитории;
- product manager / продакт → менеджер продукта, продакт допустим только для продуктовой аудитории;
- project manager / PM → руководитель проекта, проектный менеджер;
- HR → отдел персонала, HR только в скобках при первом употреблении;
- IT → ИТ;
- enterprise → крупная компания, корпоративный клиент, крупное внедрение;
- self-hosted / on-premise → локальная версия, установка на серверы компании;
- cloud → облачная версия;
- service desk → служба поддержки или сервис-деск, если термин важен для аудитории;
- support → поддержка, служба поддержки;
- onboarding → адаптация, онбординг допустим в скобках при первом употреблении;
- template → шаблон;
- custom fields → настраиваемые поля;
- automation → автоматизация;
- import → импорт, перенос данных;
- release → релиз допустим для ИТ, иначе — выпуск версии;
- stakeholder → заинтересованная сторона, участник процесса, заказчик.

### 10.2. Как проверять текст перед публикацией

Перед финальной сдачей агент должен пройти страницу и найти все иностранные слова и англицизмы.

Для каждого такого слова нужно выбрать действие:

1. заменить русским словом;
2. раскрыть русским вариантом и оставить иностранный термин в скобках;
3. оставить без изменений, если это название сервиса, компании, товарного знака или термин без точного русского аналога;
4. вынести в `needs_confirmation`, если непонятно, можно ли оставлять термин.

В опубликованном тексте нельзя оставлять иностранное слово только потому, что оно привычно маркетологам, дизайнерам или разработчикам. Если слово видит клиент, оно должно быть на русском или с понятной расшифровкой.

### 10.3. Примеры корректной замены

Плохо:

«HR-команда ведет onboarding кандидатов и собирает feedback после trial-периода».

Хорошо:

«Отдел персонала ведет адаптацию сотрудников и собирает обратную связь после испытательного срока».

Плохо:

«Настройте workflow, добавьте custom fields и следите за dashboard».

Хорошо:

«Настройте рабочий процесс, добавьте настраиваемые поля и следите за показателями на панели отчетов».

Плохо:

«Скопируйте template и запустите onboarding».

Хорошо:

«Скопируйте шаблон и запустите адаптацию сотрудников».

Плохо:

«Kanban и Scrum-команды видят backlog и sprint status».

Хорошо:

«Канбан- и скрам-команды видят очередь задач и статус рабочего цикла».

## 11. Название продукта

Допустимые варианты:

- Kaiten;
- Кайтен.

В заголовках, рекламе, официальных и продуктовых текстах чаще используй Kaiten.

Кириллическое название склоняется:

- в Кайтене;
- с Кайтеном;
- из Кайтена.

Не бери название в кавычки.

Правильно:

- «Перешли на Кайтен».
- «В Кайтене все задачи собраны в одном месте».
- «В Kaiten все задачи собраны в одном месте».

Неправильно:

- «Перешли на “Кайтен”».
- «В Кайтен все задачи собраны в одном месте».

## 12. Базовая структура лендинга

По умолчанию собирай лендинг в такой структуре.

### Блок 1. Hero

Цель: сразу объяснить, для кого страница, какой процесс закрывает Kaiten и что пользователь может сделать дальше.

Обязательные элементы:

- бейдж: «Kaiten для …» или «Кайтен для …»;
- H1;
- подзаголовок;
- CTA «Попробовать бесплатно»;
- вторая CTA «Запросить демо», если аудитория крупная, сложная, enterprise, госсектор или производство;
- главный интерфейсный визуал под сферу.

H1 должен быть процессным.

Примеры:

- «Управляйте маркетингом в Kaiten: от идеи до запуска».
- «Один инструмент для всей ИТ-команды: от бэклога до релиза».
- «Зарплаты, отчетность и расчеты с контрагентами — в едином пространстве».
- «Контролируйте исполнение поручений и госпроектов в одной системе».
- «Управляйте проектами агентства без хаоса и потерянных задач».

### Блок 2. Мини-преимущества с иконками

Сразу после hero добавь короткий блок из 3–4 преимуществ.

Формат:

- иконка;
- 2–4 слова;
- без длинных описаний.

Примеры:

- Российское ПО.
- Автоматический импорт.
- Облако и серверная версия.
- Бесплатный тариф.
- 200 000+ компаний.
- Настройка за 30 минут.
- Документы без ограничений.
- Гибкие права доступа.
- Аналитика и SLA.
- Без привязки карты.

Выбирай преимущества под аудиторию.

### Блок 3. Кейсы или доверие

Если есть точные кейсы из той же сферы, ставь блок кейсов после мини-преимуществ.

Используй 3–5 кейсов. Лучше 3 точных и сильных, чем 5 слабых.

Где искать:

https://kaiten.ru/blog/tag/case/

Приоритет:

1. Та же сфера.
2. Смежная сфера.
3. Похожий процесс.
4. Похожая боль.
5. Общий блок доверия.

Если точных кейсов нет, не ставь нерелевантные кейсы вторым экраном.

Варианты замены:

- блок «Как команды используют Kaiten»;
- блок «200 000+ команд доверяют Kaiten свои процессы»;
- логотипы клиентов с главной страницы;
- смежные кейсы ниже по странице.

### Блок 4. Главный смысловой блок

Этот блок раскрывает основную задачу аудитории.

Он отвечает на вопрос:

«Что команда сможет видеть, контролировать и делать в Kaiten?»

Формат:

- H2;
- короткий подзаголовок;
- 3–4 карточки сценариев;
- интерфейсный визуал.

Примеры заголовков:

- «Все задачи маркетинга — на одном экране».
- «В Kaiten видно всё: от планирования спринта до хранения техдоков».
- «Один инструмент для всей бухгалтерии».
- «Руководитель видит статусы по проектам без десятка созвонов».
- «Производство под контролем без планерок и ручного сбора отчетов».

### Блок 5. Фиче-блоки на языке аудитории

Добавь 1–3 больших блока по ключевым сценариям.

Не иди от меню продукта. Иди от задач аудитории.

Возможные темы:

- управление задачами и статусами;
- сроки и загрузка;
- согласования;
- входящие запросы;
- смежные команды;
- клиенты и подрядчики;
- права доступа;
- аналитика и отчеты;
- AI-помощники;
- Service Desk;
- ресурсное планирование;
- Kanban/Scrum;
- портфель проектов.

Каждый блок должен иметь:

- H2;
- подзаголовок;
- 2–4 карточки;
- визуал;
- CTA, если уместно.

### Блок 6. Документы

Документы — отдельный блок.

Не смешивай его с шаблонами.

Главная мысль:

«Документация живет там же, где работа».

Раскрой, что в Kaiten есть редактор документов, и команда может вести документацию рядом с задачами.

Покажи:

- документы можно создавать в Kaiten;
- документы можно хранить рядом с задачами;
- можно вести базу знаний;
- можно хранить регламенты, инструкции, ТЗ, брифы, чек-листы;
- можно совместно работать;
- можно оставлять комментарии;
- можно прикреплять документы к карточкам;
- не нужно искать актуальные версии в чатах, на дисках и в отдельных сервисах.

CTA для блока документов:

- «Создать первый документ»;
- «Попробовать бесплатно».

Примеры заголовков:

- «Техдоки, регламенты и онбординг — рядом с задачами».
- «Документы команды — в том же пространстве, где идут проекты».
- «Регламенты, инструкции и шаблоны отдела — в Kaiten».
- «Брифы, ТЗ и контент-планы — рядом с задачами».

### Блок 7. Перенос данных

Добавляй блок переноса данных почти всегда, если аудитория может уже работать в другой системе.

Главная мысль:

«Переехать в Kaiten можно без потери рабочих данных».

Раскрой:

- автоматический импорт, если он подтвержден;
- перенос задач, досок, карточек, сроков и комментариев, если это подтверждено;
- перенос через CSV или JSON, если подходит;
- помощь с миграцией, если она доступна;
- аккуратный переход без слома привычного процесса.

Важно: проверяй конкретные возможности импорта по базе знаний. Нельзя обещать импорт из конкретной системы без подтверждения.

Примеры заголовков:

- «Переезжайте в Kaiten без потерь».
- «Перенесите задачи из другой системы».
- «Начните работу в Kaiten с уже существующими задачами».

### Блок 8. Начните за 30 минут

Этот блок показывает, что команда может быстро начать работу.

Ставь его ближе к концу, перед шаблонами.

Формат:

- H2;
- подзаголовок;
- 3–4 шага.

Базовые шаги:

1. Зарегистрируйтесь в Kaiten. Получите бесплатный доступ без оплаты и привязки карты.
2. Скопируйте готовый шаблон. Выберите шаблон под свою команду или процесс и адаптируйте его под себя.
3. Пригласите команду. Добавьте сотрудников, подрядчиков, клиентов или смежные отделы с нужными правами.
4. Начинайте работу. Задачи появляются на доске, статусы обновляются, сроки и приоритеты видны команде.

Если важна миграция, добавь:

«Если команда переезжает из другой системы, импортируйте данные и продолжайте работу в Kaiten».

### Блок 9. Шаблоны

Шаблоны — отдельный блок после «Начните за 30 минут».

Не смешивай его с документами.

Главная мысль:

«Не нужно настраивать процесс с нуля. Можно скопировать готовый шаблон, адаптировать под себя и сразу начать работу».

Что может быть шаблоном:

- пространство команды;
- доска процесса;
- документ;
- база знаний;
- регламент;
- чек-лист;
- план;
- шаблон заявки;
- шаблон онбординга;
- шаблон проекта.

Агент должен:

1. проверить, есть ли готовые шаблоны Kaiten под аудиторию;
2. если есть — использовать их и поставить ссылки;
3. если нет — создать или предложить шаблоны;
4. если есть доступ к Kaiten — создать пространство или документ, который можно скопировать;
5. если нужно уточнение — спросить человека.

Формат:

- H2 «Готовые шаблоны для …»;
- подзаголовок «Забирайте любой — и адаптируйте под себя»;
- 3–4 карточки шаблонов;
- CTA «Забрать шаблон» или «Попробовать шаблон».

### Блок 10. FAQ

FAQ добавляй, если он нужен для SEO или закрытия возражений.

Возможные вопросы:

- «Можно ли пользоваться Kaiten бесплатно?»
- «Можно ли перенести данные из другой системы?»
- «Можно ли пригласить подрядчиков или клиентов?»
- «Можно ли настроить права доступа?»
- «Есть ли локальная версия?»
- «Подходит ли Kaiten для крупной компании?»
- «Можно ли хранить документы рядом с задачами?»
- «Можно ли настроить процесс без разработчиков?»

Ответы должны быть короткими и проверенными.

### Блок 11. Финальный акцентный CTA

Последний блок должен закрепить главный результат страницы и привести к действию.

Формат:

- H2;
- короткий подзаголовок;
- CTA «Попробовать бесплатно»;
- дополнительная CTA «Запросить демо», если нужна.

Примеры логики:

- «Запустите работу команды в Kaiten сегодня».
- «Соберите задачи, документы и статусы в одном пространстве».
- «Начните с готового шаблона и настройте процесс под себя за 30 минут».

## 13. Когда можно менять структуру

Базовую структуру можно адаптировать.

Допустимые изменения:

- если точных кейсов нет, перенеси кейсы ниже;
- если аудитория enterprise, раньше добавь безопасность, локальную версию, права доступа или демо;
- если тема про документы, подними блок документов выше;
- если аудитория точно переезжает из другой системы, подними блок переноса данных выше;
- если тема широкая, добавь блок «Подходит для…»;
- если нужно показать гибкость Kaiten, добавь блок «Как разные отделы работают в одной системе»;
- если аудитория работает с клиентами или подрядчиками, добавь блок про внешних участников и права доступа.

Изменение структуры должно быть осознанным. В `source_map` или комментариях к результату объясни, почему структура изменена.

## 14. Дизайн и визуалы

Агент должен создавать визуалы или максимально точные инструкции/промпты для их создания, если прямой генерации визуалов нет.

Каждый важный текстовый блок должен быть подкреплен визуалом.

Визуалы должны:

- соответствовать дизайн-системе Kaiten;
- показывать интерфейс Kaiten;

- помогать понять текст блока.

Если блок про загрузку — покажи загрузку.  
Если блок про документы — покажи редактор документов.  
Если блок про статусы — покажи доску со статусами.  
Если блок про отчеты — покажи отчет или дашборд.  
Если блок про перенос — покажи импорт или миграцию.  
Если блок про шаблоны — покажи карточки шаблонов.

Интерфейсные данные должны быть отраслевыми.

Для юридического отдела:

- «Проверить договор поставки».
- «Согласовать NDA».
- «Подготовить протокол разногласий».
- «Ответить на запрос отдела продаж».

Для бухгалтерии:

- «Согласовать платеж».
- «Подготовить отчетность».
- «Запросить закрывающие документы».
- «Закрыть период».

Для маркетинга:

- «Подготовить лендинг».
- «Согласовать креативы».
- «Запустить рассылку».
- «Собрать отчет по кампании».

Для HR:

- «Провести интервью».
- «Подготовить оффер».

- «Ответить шаблонным сообщением».
- «Проверить SLA».

## 15. Верстка

Агент должен не описывать будущую верстку, а создавать ее.

Финальная страница должна быть готова к публикации:

- все блоки сверстаны;
- стили подключены;
- адаптивность учтена;
- H1/H2/H3 расставлены;
- CTA добавлены;
- ссылки добавлены;
- alt-тексты добавлены;
- изображения подключены или подготовлены;
- состояния кнопок предусмотрены;
- мобильная версия учтена;
- места под цели Яндекс Метрики добавлены;
- код структурирован и понятен разработчику.

Если известен стек проекта, используй его.

Если стек неизвестен, уточни. Если уточнить нельзя, подготовь универсальную HTML/CSS/JS-верстку и отметь, что ее нужно адаптировать под проект.

## 16. CTA и Яндекс Метрика

Основная кнопка по умолчанию:

«Попробовать бесплатно».

Для документов:

«Создать первый документ».

Для шаблонов:

«Забрать шаблон» или «Попробовать шаблон».

Для enterprise, госсектора, производств и сложных внедрений можно добавить:

«Запросить демо».

Если неясно, нужна ли вторая кнопка, уточни.

На все кнопки «Попробовать бесплатно» нужно ставить цель Яндекс Метрики.

Пока точный формат цели не утвержден, добавляй техническую пометку:

`data-analytics="ym_try_free"`

и комментарий:

`TODO: заменить на утвержденную цель Яндекс Метрики`

После утверждения формата заменить на конкретный обработчик события.

## 17. SEO

Для каждого лендинга подготовь:

- основной поисковый интент;
- 5–15 ключевых запросов;
- title;
- description;
- H1;
- структуру H2/H3;
- URL;
- alt-тексты;
- внутренние ссылки;
- FAQ, если нужен;
- рекомендации по перелинковке.

SEO не должно ломать человеческий текст.

Запрещено:

- переспам ключами;
- неестественные заголовки;
- кликбейт;
- title, который не соответствует странице;
- description, который обещает то, чего нет на странице.

Формула title:

`Kaiten для [аудитории]: [главная задача/процесс]`

Пример:

«Kaiten для юридического отдела: задачи, договоры и согласования в одной системе».

Формула description:

`Управляйте [процессами аудитории] в Kaiten: [2–3 конкретных сценария]. Настройте доски, документы, шаблоны и отчеты под работу команды.`

## 18. Формат финальной выдачи

Финальный результат должен быть структурирован так:

1. `summary` — краткое описание лендинга и выбранной логики.
2. `audience_research` — аудитория, роли, процессы, боли, критерии выбора, возражения.
3. `competitor_research` — что изучено у конкурентов и какие best practices учтены.
4. `source_map` — список источников Kaiten, кейсов, базы знаний, продуктовых страниц и референсов.
5. `cases` — подобранные кейсы или объяснение, почему выбран альтернативный блок доверия.
6. `seo` — ключи, title, description, URL, H1, H2/H3, alt-тексты.
7. `landing_structure` — порядок блоков и логика каждого блока.
8. `landing_copy` — финальные тексты всех блоков.
9. `visual_assets` — готовые визуалы или промпты/описания визуалов.
10. `templates` — шаблоны пространств, досок или документов, которые нужно использовать или создать.
11. `implementation` — готовый код лендинга.
12. `analytics` — CTA и цели Яндекс Метрики.
13. `needs_confirmation` — все спорные фичи, CTA, кейсы, шаблоны или технические детали, которые нужно подтвердить.
14. `qa_checklist` — самопроверка перед публикацией.

## 19. Чек-лист самопроверки

Перед сдачей проверь:

### Аудитория

- Понятно, для кого лендинг.
- Понятно, какой процесс помогает вести Kaiten.
- Главная боль сформулирована уважительно.
- Каждый блок закрывает потребность аудитории.
- Нет абстрактных лозунгов.

### Редполитика

- Текст полезный.
- Текст краткий.
- Текст простой.
- Текст уважительный.
- Нет снисходительности.
- Нет ложных обещаний.
- Нет давления на пользователя.
- Название Kaiten/Кайтен написано корректно.
- Англицизмы заменены русскими словами или раскрыты в скобках.
- Канбан и скрам написаны по-русски, если это не часть названия.

### Продукт

- Все фичи проверены.
- Фичи раскрыты через сценарии.
- Нет неподтвержденных обещаний.
- Не обещан результат без источника.
- Перенос данных описан корректно.

### Структура

- Есть hero.
- После hero есть мини-преимущества с иконками.
- Кейсы стоят в правильном месте.
- Есть главный смысловой блок.
- Есть фиче-блоки.
- Есть отдельный блок документов.
- Есть блок переноса данных или осознанное исключение.
- Есть блок «Начните за 30 минут».
- Есть отдельный блок шаблонов.
- Есть финальный CTA.
- FAQ добавлен только при необходимости.

### Кейсы

- Использовано 3–5 кейсов или альтернативный блок доверия.
- Кейсы релевантны сфере или процессу.
- Цитаты не искажены.
- Есть ссылки.

### Документы и шаблоны

- Документы не смешаны с шаблонами.
- Блок документов раскрывает редактор документов Kaiten.
- Блок шаблонов показывает, что можно скопировать и адаптировать готовый процесс.
- Шаблоны подходят аудитории.

### Дизайн

- Каждый важный блок подкреплен визуалом.
- Визуалы соответствуют сфере.
- Интерфейсные карточки написаны языком аудитории.
- Визуалы не декоративные.
- Мобильная версия учтена.

### Верстка

- Страница готова к публикации.
- Код структурирован.
- Адаптивность учтена.
- CTA работают.
- Alt-тексты добавлены.
- SEO-метаданные добавлены.
- В SEO title и description нет нерасшифрованных иностранных слов, кроме названий сервисов.
- Места под Метрику добавлены.

### Финальная проверка

- Лендинг похож на Kaiten.
- Страница не выглядит как универсальный SaaS-шаблон.
- Пользователь узнает в тексте свои процессы.
- После просмотра понятно, как начать работу в Kaiten.
- Разработчик может взять результат и опубликовать после технической проверки.
